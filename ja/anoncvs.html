Anonymous CVS とは ?

Anonymous CVS は、現時点での OpenBSD ソースに対して加えられた更新に関して、 OpenBSD のソースツリーのあなたのローカルなコピーを最新に維持しておくための 方法だ。開発の最先端に追従することに加え、リリースの errata のためのパッチ を追跡することもまた可能です。

他のソースコード更新技術に対する Anonymous CVS の主な利点は、中央の ソースコードリポジトリかミラーに対して、それが直接働くということ だろう。これはつまり、あなたの変更と他のソースの変更に関する更新や マージを制御したり、diff を取ったり、変更履歴や他のことを中央の リポジトリに対して質問したりするために、フルセットの CVS コマンドを あなたは使用可能だということだ。

OpenBSD プロジェクトは、現状では、次の 5 個の主ソースリポジトリを持つ。

  • src - OpenBSD オペレーティングシステムの全ソースコードを収容。
  • ports - OpenBSD Ports を収容。
  • www - すべての OpenBSD の Web ページ (このページを含む) を収容。
  • x11 - OpenBSD 対応の XFree86-3 ソフトウェアプロジェクトを収容。
  • xf4 - OpenBSD 対応の XFree86-4 ソフトウェアプロジェクトを収容。

要約すると、Anonymous CVS を使用することの実際の強みは、それが「寛容な」 ソースコードコントロールシステムであるということであり、それは、あなたが そのまま続けずに解決しなければならない難問のリストを放置するのではなく、 むしろ、あなたのローカルなソースに対してあなたが加えた変更と、あなたの 全体のソースツリーを更新するための「最善の努力」をする ことを尊重する。

CVS とは ?

CVS とは、OpenBSD のソースツリーを管理するのに使用されているソースコードコントロール システムである。それは、すべての正式にリリースされたソースコードとそれに対する 変更のための中央のリポジトリを実装する一方で、開発者に対しては彼らの作業中の変更を 含むソースコードのローカルなコピーを保守することを許している。「書き込み権」 を持つ開発者は、OpenBSD のソースツリーに対して直接変更をコミットできる一方で、 「Anonymous CVS」のユーザは「読み出し権」を持ち、そして、ソースの自分の ローカルなコピーを最新の状態に維持でき、中央のリポジトリに対する質問を発行できる。

CVS の主な強みは、あなたの行ったあなたのローカルなコピーに対する変更とともに、 中央のリポジトリに対する変更のインテリジェントなマージを実行可能なことだろう。 これは、もし、あなたがモジュールに対して変更を加えて、更新を実行した場合にも、 あなたの変更が「吹き飛ばされる」ようなことがないことを意味しており、むしろ、 CVS は、中央のソースに対する変更とあなたがあなたのローカルなコピーに対して 加えた変更とをマージするよう、最大限の努力を払おうとする。

変更を完全にマージできなかった場合には、CVS は「柔軟なフォールバック」を提供し、 また、あなたのローカルなコピーに対する注釈された変更を提供し、あなたのバージョンの 変更されていないコピーを保存し、そして、あなたの要求した他のソースモジュールの 更新を継続する。

Anonymous CVS を使い始めるために

最新版の CVS は、 Cyclic にある。 バージョン 1.6 以前のものは推奨しないし、また、動作しないだろう。 もし、OpenBSD を既にインストールしていれば、CVS は含まれている。

OpenBSD CD を持つ人は、その中に CVS/ ディレクトリがあるのを見たことがあるかも 知れない。実際、これには理由があって、CD には、更新するのに便えるよう、OpenBSD の src モジュールのチェックアウトされたものが入っている。このツリーを使うのは、 OpenBSD のソースツリーの新鮮なチェックアウトより、格段に速く CVS の最初の更新ができる だろう。ここで、CD の利用法には 2 通りある。

  • CD から /usr/src に CVS ツリーをコピーする方法 (CD は /mnt にマウントされているものと仮定) 
    	# cd /mnt; pax -rw CVS Makefile [a-z]* /usr/src
  • 書き込み可能なディレクトリの下に CD を使用した union マウントを使用 (mount_union(8) を参照) (この方法は、ツリーからコンパイルする場合にのみ使用可能) 
    	# mount -t union -o -b /mnt /usr/src

手元に CD を持たない人に対しては、ソースリポジトリを「チェックアウト」するために、 cvs(1) を使うことができる。これについては、次節で議論しよう。

その後、/usr/src は、良きチェックアウトのための領域となり、ここで、すべての cvs(1) コマンドがうまく動くことだろう。

ソースツリーの取得・更新に CVS を使おう

CVS は、ソースを受け取り、それを更新するための単純な方法であることを目指して 設計されているので、何をするにもそれほど難解ではない。最初に、-current を追いかけたいのかパッチブランチに追従したいのかを決定するだけだ。 -current ツリーには細かい変更のすべてが含まれており、 パッチブランチは、正式リリースに加え、errata から のパッチが既に適用されたものとなっている。 -current の定義については、 更新ミニ FAQを参照して欲しい。

いったん、追いかけるツリーを決めたら、どの Anonymous CVS サーバを 使うのかを選択しなければならない。これらのサーバのリストは、 以下のとおり。しかしながら、これらのサーバへの アクセス方法には 3 通りある。

ssh
Secure Shell を Anonymous CVS サーバへのアクセスに使うことができる。 この方法は暗号化されるので、お薦めの方法だ。 2.6 の時点で、OpenBSD は標準ディストリビューションとして OpenSSH を含んでいる。
rsh
Remote Shell は、ssh での アクセスができないユーザのために、いくつかのサーバ上で使うことができる。
pserver
pserver 他のふたつの接続を阻むファイアウォールの後ろにいる ユーザに対しては、最も役に立つものだろう。

注: rsh を使いたいというユーザは、最初に CVS_RSH 環境変数を rsh にセットする必要がある。

  • Korn/Bourne シェルの場合: 
    	$ export CVS_RSH=/usr/bin/rsh
  • csh/tcsh の場合: 
    	% setenv CVS_RSH /usr/bin/rsh

いったん、どの Anonymous CVS サーバを使うのかを、そして、 どの方法を使用するのかを選択したなら、cvs を使い始めることができる。 CD を持っている人は、自分のシステムにソースを取得するために、 上記の方法で CD 上の CVS のチェックアウトで cvs を使い始められる。 もし、CD を持っていないなら、ソースをチェックアウトするために次の方法を使おう。 この方法は、OpenBSD のソースを /usr/src に置くためのものだ。 

	# cd /usr; cvs checkout -P src

上記の方法は、-current のソースツリーをチェックアウトすることになる。多くの人に とっては、パッチブランチのソースが欲しいことだろう。このパッチブランチをチェックアウト するためには、次の例のようにコマンドライン上でタグを指定する必要がある。 

	# cd /usr; cvs checkout -P -rOPENBSD_2_9 src

これは、2.8 の場合なら、OPENBSD_2_8 となる。

現時点では、OPENBSD_2_9 のタグのみが、リリースのソースと適用済の erratta を含む。

利用可能な Anonymous CVS サーバ

ふたつのレベルのソースツリーへのアクセス方法がある。

開発者向けの読み書きアクセス:
ソースツリーへの変更を commit する必要のある開発者は、OpenBSD のマシン上に アカウントを持つ必要がある。このアクセス方法を得るということは、OpenBSD の 他の開発者とともにソースに対して作業するという、自然な結果となろう。 誰かが何か良い仕事をし、そして、それをチームで作業できるようにした場合に、 彼らはアカウントを取得することになるだろう。
すべての人向けの読み込み専用アクセス:
読み込み専用の CVS リポジトリに対しては誰でもアクセスすることができる。 これらの読み込み専用の CVS リポジトリのコピーは、ミラーされていることも多い。 それらを使用するには、CVSROOT 環境変数に次の値のどれかひとつを セットして欲しい。

cvs(1) を使っての暗号ソースの取得

重要な注意事項: 誰もが知っておくべきこととして、暗号ソフトウェアに関しては 若干の問題がある。

  • OpenBSD のソースはカナダからのもの なので、 カナダの個人によって研究されたものとして、 また、 カナダの輸出制限リストの記述にあるように、 カナダから世界に向けての暗号ソフトウェアの輸出は合法的なものだ。

  • しかしながら、あなたが、米国やカナダ以外の地域に住んでいるのなら、 あなたは米国内にある Anonymous CVS サーバから OpenBSD のソースの 暗号セクションを持っていくべきではない。問題のファイルは次のとおり。
    • src/kerberosIV/*
    • src/kerberosV/*
    • src/lib/libdes/*
    • src/lib/libc/crypt/crypt.c
    • src/lib/libc/crypt/morecrypt.c
    • src/sys/crypto
    • src/sys/netinet
    • src/usr.sbin/afs/src/rxkad/*
    • XF4/xc-mit/lib/Xdmcp/Wraphelp.c
    • XF4/xc-old/lib/Xdmcp/Wraphelp.c
    • XF4/xc/lib/Xdmcp/Wraphelp.c
    米国の ITAR 軍需品リストによって、暗号ソフトウェアは、 米国からカナダだけに対して輸出されたものかも知れないのだ。

  • OpenBSD プロジェクトでは、さらなる Anonymous CVS サーバを探している。 あなたがどのように支援できるかを知るためにも読んでいただきたい。

cvs(1) の使用例

注意: もし、(パッチブランチのような) ブランチを -current に更新したい場合、 -A フラグを cvs に追加することができるが、このフラグは他にはほとんど 使い道がないものだ。古目のバージョンのバージョンの OpenBSD のドキュメンテーションでは、 多くの例の中で、このフラグの使用が推奨されている。我々は最早、絶対に必要でない限りは、 このフラグを推奨しない。

Anonymous CVS サーバの使用法の例は次のとおり。 

% setenv CVSROOT anoncvs@anoncvs.ca.openbsd.org:/cvs
% cd /tmp
% cvs get src/sys/arch/sparc
    [リポジトリからあなたのマシンへファイルをコピー]
% cvs log src/sys/arch/sparc/sparc/locore.s
    [選択したファイルのコミットログを表示]
% cvs diff -bc -r1.1 -r1.5 src/sys/arch/sparc/sparc/locore.s
    [リビジョン 1.1 とリビジョン 1.5 との間の変更点を表示]

cvs を ``pserver'' (ssh や rsh を使用する代わりの直接の TCP 接続) instead of using ssh or rsh) で使用する場合には、いったん login しなければならない。 

% setenv CVSROOT :pserver:anoncvs@anoncvs.ca.openbsd.org:/cvs
% cvs login
(anoncvs@anoncvs1.ca.openbsd.org への login)
CVS password: anoncvs
    [この行を ~/.cvspass に書く (ファイル名は CVS_PASSFILE がオーバーライドする)。]
    [上記において、'blah' とタイプした後の私の ~/.cvspass 内の行の一例            ]
    [password is:                                                                 ]
    [:pserver:anoncvs@anoncvs5.usa.openbsd.org:/cvs Au'yc                         ]
    [一度 login した後は、他のすべての上記の CVSROOT が動くようになるだろう。     ]
% cvs get ksrc-i386 ksrc-common
    [あなたは、i386 カーネルを再構築するのに必要なものだけ、受け取ることができる。]

Anonymous CVS を使用する人が、普通に自分のソースツリーを更新 するには、

  • まず、初期のツリーを `get' する。

    (もし、-current を追いかけているのなら): 

    	# setenv CVSROOT anoncvs@anoncvs.ca.openbsd.org:/cvs
	# cd /usr
	# cvs -q get -P src

    (もし、2.9 のパッチブランチを追いかけているのなら): 

    	# setenv CVSROOT anoncvs@anoncvs.ca.openbsd.org:/cvs
	# cd /usr
	# cvs -q get -rOPENBSD_2_9 -P src
  • その後、いつでもこのツリーを `update' するには:

    (もし、-current を追いかけているのなら): 

    	# cd /usr/src
	# cvs -q up -Pd

    (もし、2.9 のパッチブランチを追いかけているのなら): 

    	# cd /usr/src
	# cvs -q up -rOPENBSD_2_9 -Pd
    これを実行するごとに、いつでも自分の /usr/src ツリーを同期させるだろう。これに よってあなたのローカルな変更が壊されることはなく、むしろ、変更をマージしようと 試みる。もし、obj ディレクトリ (obj シンボリックリンクではない) を使用している のなら、cvs のコマンドラインに、"-I obj" を追加しようと思うかも知れないが、これ は、リポジトリの中にはない obj ディレクトリが現れるたびに出力されるウォーニング が出力されないように cvs を動かすことができるようになるだろう。

  • 注: もし、最初に異なるサーバから持ってきた、あるいは CD から持ってきたソースツリー を更新しようとするのなら、-d $CVSROOT オプションを cvs のコマンドライン に追加する必要がある。 
    	# cd /usr/src
	# cvs -d $CVSROOT -q up -Pd

ports を使うのも、src と同様だ。

  • (もし、-current を追いかけているのなら): 

    	# setenv CVSROOT anoncvs@anoncvs.ca.openbsd.org:/cvs
	# cd /usr
	# cvs -q get -P ports

    • (もし、2.9 のパッチブランチを追いかけているのなら): 

    	# setenv CVSROOT anoncvs@anoncvs.ca.openbsd.org:/cvs
	# cd /usr
	# cvs -q get -rOPENBSD_2_9 -P ports
  • その後、いつでもこのツリーを `update' するには:

    (もし、-current を追いかけているのなら): 

    	# cd /usr
	# cvs -q up -Pd ports

    (もし、2.9 のパッチブランチを追いかけているのなら): 

    	# cd /usr
	# cvs -q up -rOPENBSD_2_9 ports
上記の例において、cvs の出力を最小化させるために、-q オプションも 指定可能だ。画面いっぱいの出力がお好みの向きには、これのオプションは 外しておいた方が幸せだろう。

また、ローカルにパッチを当てたモジュール (ここでは cd.c) をバグレポートに 含めるために diff を取るには、 

	# cd /usr
	# cvs diff -u src/sys/scsi/cd.c > /tmp/patch

(CVS ソースに含まれている) cvs(1) マニュアルページには、 もっと多くの CVS の使い方についての情報がある。

X11 ソースツリー

Anonymous CVS はまた、OpenBSD の X11 (XFree86) ソースツリーもミラーしている。二枚目の CD からの XFree86 ソースツリーを更新するために、上記のレシピを適用することができる。 X11 の cvs モジュールで XFree86 3 のソースツリーをコピーしたり、ダウンロードしたり できるが、XFree86 4 を含んだ現時点でのモジュールは XF4 となっている。 現状では、XF4 のソースを /usr/XF4 に取得するのに、ふたつの方法がある。

  • CD からツリーをコピーする (二枚目の CD は /mnt にマウントされているものと仮定)。 
            # cd /mnt; cp -Rp XF4 /usr
  • 書き込み可能なディレクトリの下に CD を使用した union マウントを使用する。 
            # mount -t union -o -b /mnt/XF4 /usr/XF4
この後、/usr/XF4 は cvs によって既に使われていることだろう。例としては、 たとえば、それを -current ソース (既に CVSROOT 環境変数としてセット していると仮定) に更新することができる。 
        # cd /usr/XF4
        # cvs -q update -Pd

警告: cvs を使用する場合には、カレントディレクトリが、参照しているツリーの root か、 /tmp のような独立した場所かのどちらかになっているよう、注意すべきであろう。 "get" のようなあコマンドの場合、カレントディレクトリの下に、任意のサブツリーを 作成できてしまうので、これに続く更新は再帰的にこのサブツリーを更新していくだろう。

Anonymous CVS サービスは、経験の浅い開発者にとって、CVS のオペレーションを 学び、"commit" アクセスを経験するまでの、とても複雑な開発プロセスを経験するための 絶好のチャンスを提供しているが、高品質で有用な技術を見せることの結果として、 彼らはいずれ自然に開発者としてのアクセスを与えられるようになるだろう。 同様に、パッチを提供する人は、CVS ツリーに関係する自分の "diff" を作成することが できるが、それは、簡単に統合できるものだろう。

rsh(1) と ssh(1) をどちらを使うか ?

デフォルトで、CVS クライアントは、CVS サーバとのやり取りに ssh ("secure shell": OpenSSH) を使用する。

多くの CVS サイトは、最早セキュリティ上の理由から rsh をサポートしない。 ファイアウォール等のローカルな問題や、slirp のような不完全なプロトコルエミュレータ もまた、rsh の利用の邪魔になるかも知れない。しかし、それでも、もし、rsh を使う必要が ある場合には、CVS_RSH 環境変数に rsh を指定しなければならない。 (典型的には /usr/bin/rsh).

もし、ローカルなポリシーが ssh のデフォルトポートの 22 番の外との接続を邪魔する場合、 ポート 2022 がその代わりに使用されるかも知れない。しかしながら、それはすべての Anonymous CVS サーバがこのポートでの ssh 接続を受け付けるわけではないことに注意する必要がある。 さらに、セキュリティ上の理由から、最近のバージョンの ssh では、暗号なし が無効化されているので、ほとんどの Anonymous CVS サーバは最早の接続を受け付けない。 また、CVS では既に圧縮が有効なので、ssh での圧縮を有効にしようとはしない。

上記に述べた制限や落し穴を避けるために、 $HOME/.ssh/config コンフィグレーションファイルなどを 指定することができる。 

	Host anoncvs.ca.openbsd.org
	    Compression no
	    Port 2022

CVS をスタートする際は、少々騒々しいので、少し静かにさせるために、 次のようなことをしたくなるかも知れない。 

	% setenv CVS_CLIENT_PORT -1

sup(1) を使った CVS リポジトリのミラーリング

OpenBSD のツリーそのものをミラーしようとするユーザは、 anoncvs.usa.openbsd.org や anoncvs1.usa.openbsd.org (これらは異なるマシンだ) から、そうしようとするかも知れない。しかし、これらは cvs ツリーであり、ソースツリーをチェックアウトしているのではないことに 注意する必要がある。これは、もし、高速な cvs オペレーション (diff、annotate 等) や、もし、複数のソースツリーを持ち、新しいものを転送したい (そうすれば、ローカルな cvs ミラーからツリーをチェックアウトすることができる) 場合にのみ、有用でしかない。

supfile の例は次のとおり。 

	cvs host=anoncvs.usa.openbsd.org hostbase=/ base=/home delete

これは、/home/sup の中の最後の sup データファイルとともに cvs ツリーを /home/cvs にミラーするだろう。すべての OpenBSD cvs ツリーは、現状では およそ 1.3GB のサイズになる。

anoncvs ミラーの構築

もし、あなたが Anonymous CVS のミラーサイトに作ろうと思うのなら、Anonymous CVS のメンテナに連絡を取っていただきたい。 Anonymous CVS ミラーは約 1.2GB のディスク容量を必要とし、Anonymous CVS ユーザ ひとり当たり 32MB までのスワップを使用する (ユーザは大きなオペレーションを行う ものと仮定する一方で、小さなオペレーションはより少量のリソースを使用すると仮定 するが、それでもなお、 Anonymous CVS は ftp や sup より大きなインパクトがある)。 そのような Anonymous CVS マシンは、サービスを行おうとする地域に対する優れた ネットワーク接続を持つべきだろう。Anonymous CVS サーバを構築について記述された ドキュメントが利用可能となっている。

最終的な注意事項

ソースツリーをアップグレードした後、構築しようとする前に、 /usr/src/Makefile の先頭のコメントを読むべきだろう。 また、もし、可能なら、make build を実行する前に 新しいカーネルを構築しておくべきだ。場合によっては、カーネルを構築できる ようになる前に、configユーティリティを再構築し、インストールして おく必要があるかも知れない。もし、config GENERIC が失敗するようなら、 たぶん、このケースだろう。

ソースからの再構築によって、リリースから -current ツリーへのアップグレードは、 しばしば明確ではない依存性のため、どちらかと言えば難しいということに注意するのは、 重要なことだろう。なので、ソースからツリーを構築しようとする前に、まず、最新の スナップショットをインストールすることを提案しておこう。
