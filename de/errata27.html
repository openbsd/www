Dies ist die OpenBSD Version 2.7 errata & Patch Liste:

OpenBSD patch branch Informationen gibt es hier.

Die Errata für eine bestimmte Version findest du hier:
2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.8, 3.0, 3.1, 3.2, 3.3, 3.4.
Du kannst auch eine tar.gz Datei holen, die all die folgenden patches enthält. Die Datei wird einmal am Tag erneuert.

Die Patches weiter unten sind mit CVS im OPENBSD_2_7 Patch Branch erhältlich.

Alle Architekturen

  • 024: SECURITY FIX: 18. März 2001
    Die readline library, die mit OpenBSD ausgeliefert wurde, erlaubt das Erzeugen von history files mit einer 'permissive' umask(2). Das kann zu einem Leck an sensitiven Informationen in Applikationen führen, die Passwörter und ähnliches während der Userinteraktion benutzen. (ein Beispiel ist mysql). Zusätzlich wird das momentane Arbeitsverzeichnis benutzt, wenn die HOME Umgebungsvariable nicht gesetzt ist; dieser Patch schaltet die HISTORY Datei ab, wenn HOME nicht gesetzt ist.
    Es gibt einen Source-Code Patch, der das Problem beseitigt.

    Es gibt einen exploitbaren heap corruption bug in sudo.
    Es gibt einen source code Patch, der das Problem beseitigt.

  • 038: SECURITY FIX: 29. Jan. 2001
    Den named mit ISC BIND 4.9.8-REL erweitern, was einige 'buffer vulnerabilities' verhindert.
    Es gibt einen Source-Code-Patch, der das Problem beseitigt.

  • 037: SECURITY FIX: 4. Dezember 2000
    OpenBSD 2.7's ftpd enthält einen 'one-byte overflow' in der 'replydirname()' Funktion.
    Es gibt einen Source-Code-Patch, der das Problem beseitigt.

  • 035: SECURITY FIX: 10. Nov 2000
    Andere Server können OpenSSH Clients dazu bringen Agent oder X11 forwarding zu machen. Dieses Problem ist seit OpenSSH 2.3.0 behoben.
    Es gibt einen Source-Code-Patch, der das Problem beseitigt.

  • 033: RELIABILITY FIX: 6. Nov. 2000
    Ungültige Felder im exec header können einen crash verursachen.
    Es gibt einen source code Patch, der das Problem löst.

  • 032: SECURITY FIX: 26. Okt 2000
    Es gitb zwei möglicherwiese 'exploitbare' (exploitable) 'Buffer overflows' in den X11 libraries, die den xtrans Code benutzen. Einer dieser wunden Punkte wurde an die BUGTRAQ Mailingliste gemeldet.
    Es gibt einen source code Patch, der das Problem löst.

  • 031: SECURITY FIX: 18. Okt 2000
    Apache hat einige Bugs in mod_rewrite und mod_vhost_alias, die dazu führen, dass willkürliche Dateien, die für den 'www' User auf dem Server lesbar sind, unter bestimmten Umständen auch für andere User offenliegen, falls diese Module benutzt werden. (Diese Module sind aber standardmässig nicht aktiviert).
    Es gibt einen source code Patch, der das Problem löst.

  • 030: SECURITY FIX: 10. Okt. 2000
    Der telnet Daemon wertet die TERMINFO, TERMINFO_DIRS, TERMPATH und TERMCAP (wenn sie mit '/' beginnen) Umgebungsvariablen nicht aus.
    Es gibt einen source code Patch, der das Problem löst.

  • 029: RELIABILITY FIX: 9. Okt. 2000
    Es gibt einen nicht-ausnutzbaren 'Buffer Overflow' (non-exploitable) im Testmodus von sendmail.
    Es gibt einen source code Patch, der das Problem löst.

  • 028: SECURITY FIX: 6. Okt. 2000
    es gibt einige 'printf format string' Bugs in verschiedenen privilegierten Programmen.
    Es gibt einen source code Patch, der das Problem löst.

  • 027: SECURITY FIX: 6. Okt. 2000
    libcurses benutzt Terminal Beschreibungen sowohl im $HOME/.terminfo Verzeichnis, als auch in der TERMCAP Umgebungsvariable für setuid und setgid Applikationen.
    Es gibt einen source code Patch, der das Problem löst.

  • 026: SECURITY FIX: 6. Okt. 2000
    Es gibt einen 'format string' bug in talkd(8). Es ist jedoch unklar, wie die Folgen aussehen.
    Es gibt einen source code Patch, der das Problem löst.

  • 025: SECURITY FIX: 3. Oktober 2000
    Es gibt eine 'format string' Verwundbarkeit in der pw_error(3) Funktion. Das manifestiert sich in einem Sicherheitsloch im chpass Werkzeug (Utility). Um das ganze zu umgehen, tippe 
    # chmod u-s /usr/bin/chpass
    Es gibt einen source code Patch, der das Problem löst.

  • 024: SECURITY FIX: 18. Sep 2000
    Falsche ESP/AH Pakete können unter bestimmten Umständen einen Crash verursachen. Es gibt einen source code Patch, der das Problem löst.

  • 023: SECURITY FIX: 16. Aug 2000
    Es gibt eine 'format string' Verwundbarkeit in xlock. Als Übergangslösung, die seine Funktion einfach abschaltet, tue folgendes: 
    # chmod u-s /usr/X11R6/bin/xlock

    Es gibt einen source code Patch, der das Problem löst.

  • 021: SECURITY FIX: 14. Juli 2000
    Verschiedene Probleme in den X11 libraries haben verschiedene Seiteneffekte. Wir bieten einen Jumbo Patch, um sie zu beseitigen.

    • Garstiger X Server Dos
      Dies wird vom Patch für xc/programs/Xsever/os/secauth.c abgedeckt.
    • Verschiedene garstige libX11 Löcher
      Dies wird von den Patches für xc/lib/X11 abgedeckt.
    • libICE DoS
      Dies wird von den Patches für xc/lib/ICE abgedeckt.
    • Server overflow
      Dies wird von den Patches für xc/programs/Xserver/xkb abgedeckt.
      Die OpenBSD Standardinstallation ist davon nicht betroffen, da der XFree86 Xwrapper bereits Tests gegen unzulässige Parameter enthält.

    Es gibt einen source code Patch, der das Problem löst.
    Hinweis 1: tcl/tk wird benötigt, um X11 aus dem source zu erzeugen (zu kompilieren).
    Hinweis 2: Beim 're-building' musst du das Kommando `make DESTDIR=/ build' oder es wird einen Fehler im letzten Schritt des build (makewhatis) geben. Wenn du das Setzen des DESTDIR vergisst, kannst du den 'build error' ignorieren. Die 'whatis' Datenbank wird dann beim nächsten Durchlauf von von /etc/weekly neu erzeugt.

mac68k

  • 022: INSTALLATION FIX: 14. Juli 2000
    Der MacOS Installer, der mit OpenBSD 2.7 ausgeliefert wird, erzeugt nicht alle notwendigen Devices, insbesondere das /dev/arandom Device ,dass für Userland Crypto wie z.B. ssh gebraucht wird, wird nicht erzeugt. Das Problem tritt auf, wenn ssh-keygen beim Erzeugen von RSA oder DSA Schlüsseln versagt, was zu Fehlermeldungen wie RSA-generate_keys failed oder DSA-generate_keys failed führt.
    Um das ganze zum Umgehen, führe die folgenden Kommandos als root aus, sobald die Maschine installiert ist. 
    # cd /dev
# ./MAKEDEV arandom

    Nach diesen Kommandos (und vermutlich nach der Installation eines der ssl27-Pakete), reboote deine Maschine und sie wird ganz normal ssh Schlüssel erzeugen.
