Alle Architekturen

  • 036: SECURITY FIX: 14. November 2002
    Ein Pufferüberlauf im named(8) kann einem Angreifer erlauben, Code mit den Privilegien des named auszuführen. Auf OpenBSD läuft der named nicht als root in einer ,chrooted' Umgebung, was den Effekt dieses Fehlers mildert.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 035: SECURITY FIX: 6. November 2002
    Ein nicht korrektes Prüfen der Argumente im Systemaufruf getrlimit(2) kann es einem Angreifer erlauben, den Kernel zum Absturz zu bringen.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 034: SECURITY FIX: 6. November 2002
    Ein Angreifer kann die Beschränkungen von sendmails restriktiver Shell smrsh(8) umgehen und beliebige Befehle mit seinen eigenen Privilegien ausführen.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 033: SECURITY FIX: 21. Oktober 2002

  • 032: SECURITY FIX: 7. Oktober 2002

  • 031: SECURITY FIX: 11. August 2002

    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 013: SECURITY FIX: 30. Juli 2002

  • 016: SECURITY FIX: 19. März 2002
    Unter bestimmenten Umständen ist es auf Systemen, die YP mit netgroups in der password-Datenbank benutzen, für die Daemons rexecd(8) und rshd(8) möglich, die Shell aus dem Passworteintrag eines anderen Benutzers auszuführen.   • 015: RELIABILITY FIX: 13. März 2002
    Unter bestimmten Umständen kann die Kompressionsbibliothek zlib dynamisch zugewiesenen Speicher doppelt freigeben. Unter OpenBSD ist dies kein Sicherheitsproblem, da die BSD-free(3)-Funktion das entdeckt. Es gibt auch eine Zlib-Kernelkomponente, die von pppd und IPsec genutzt werden kann. Es gibt zur Zeit keine bekannte Möglichkeit, den Kernel anzugreifen.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 011: SECURITY FIX: 17. Januar 2002
    Wenn die Sendmail-Alternative Postfix auf einem System installiert ist, kann es einem Angreifer gelingen, auf dem lokalen Rechner root-Rechte mittels sudo(8) zu erlangen. sudo ruft den Mailer als root mit einer Umgebung auf, die vom aufrufenden Benutzer stammt. Obwohl das ein Fehler in sudo ist, ist er nach unserer Meinung nicht ausnutzbar, wenn Sendmail (der Mailer, der mit OpenBSD ausgeliefert wird) benutzt wird. Mit Version 1.6.5 übergibt sudo dem Mailer eine Umgebung, die nicht mehr vom aufrufenden Benutzer beeinflusst werden kann.
    Es existiert ein Quelltextpatch, der das Problem behebt.

  • 002: SECURITY FIX: 12. November 2001
    sshd(8) wird von OpenSSH 3.0 auf OpenSSH 3.0.2 geupgradet, um die folgenden Probleme zu beheben:

    • Eine Sicherheitslücke, die einem Angreifer erlauben könnte, sich teilweise zu authentifizieren, wenn - und nur wenn - der Administrator KerberosV eingeschaltet hat.
      Standardmäßig kann die KerberosV-Unterstützung von OpenSSH nur aktiviert werden, wenn KerberosV zuvor sauber konfiguriert wurde.

    • Ein exzessiver ,memory clearing'-Fehler (von dem wir glauben, dass er nicht auszunutzen ist) existiert auch, aber da das auch zu einem Absturz des Daemons führen kann, bieten wir hierfür ebenfalls einen Patch an.

    • Eine Verwundbarkeit in der Übergabe der Umgebungsvariablen in der UseLogin-Option von sshd.

    • Verschiedene andere unkritische Fixes.

    Effektiv ein Upgrade von OpenSSH 3.0 auf OpenSSH 3.0.2. Es existiert ein Quelltextpatch, der das Problem behebt. Das ist die zweite Version des Patches.

  • 001: INSTALL ISSUE: 12. November 2001
    Ein kleiner Fehler im Installationsskript führt dazu, dass die Datei /etc/hosts falsch angelegt wird.
    Die resultierende Datei enthält eine Zeile wie die folgende:

       #.#.#.# hostname. hostname

    Die Zeile sollte eigentlich wie folgt aussehen:

       #.#.#.# hostname.domainname.com hostname

    Um das Problem zu beheben, editiere einfach die Datei und füge den Domänennamen am passenden Platz ein.
