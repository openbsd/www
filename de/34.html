Die Dateien dieser Version erhältst du auf einen der folgenden Wege:
  • Bestelle eine CDROM über unser Bestellsystem.
  • Sieh dir die FTP Seite an, um eine Liste der FTP-Mirrors zu betrachten, und den passenden für dich herauszufinden.
  • Gehe in das Verzeichnis pub/OpenBSD/3.4/ auf einem der Mirrors deiner Wahl.
  • Sieh dir den Rest dieser Seite durch.
  • Schau dir die 3.4 Errata Seite an, ob es Bugs und/oder Workarounds gibt.
  • Lies dir das detaillierte Log der Änderungen zwischen den Versionen 3.3 und 3.4 durch.


Hinweis: Alle anwendbaren Copyrights und Danksagungen können in den passenden Quellcodedateien in src.tar.gz, sys.tar.gz, XF4.tar.gz oder den Dateien gefunden werden, die via ports.tar.gz geholt wurden. Die Distributionsdateien, die man braucht, um mit Hilfe der Datei ports.tar.gz Pakete bauen zu können, sind aus Platzgründen nicht mit auf den CDROMs.

Was gibt es Neues?

Dies ist eine unvollständige Liste von neuen Features und Teilsystemen, die in OpenBSD 3.4 integriert sind. Eine vollständige Liste findet sich im changelog, das zu 3.4 führt.

  • Die i386-Architektur wurde auf das ELF executable format umgestellt.

  • Weitere W^X Verbesserungen, unter anderem Unterstützung für die i386-Architektur. Bei nativen i386 binaries wurde das Codesegment umgeordnet, um die Trennung von Code und Daten voranzutreiben, und das CPU Codesegment-Limit wird dazu benutzt, eine bestmögliche Lösung für die Ausführung von Code zu gewährleisten.

  • ld.so(1) auf ELF Plattformen lädt Bibliotheken jetzt in einer zufälligen Reihenfollge, um widerstandsfähiger gegen Attacken zu sein. Die i386-Architektur bildet ebenfalls Bibliotheken auf einigermaßen randomisierten Adressen ab. Zusammen mit W^X und ProPolice erhöhen diese Änderungen die Schwierigkeit, einen Fehler in einer Anwendung (wie einen Buffer Overflow) erfolgreich auszubeuten.

  • Ein "static bounds checker" wurde zum Compiler hinzugefügt, um grundlegende Überprüfungen von Funktionen durchzuführen, die Puffer und Größenangaben akzeptieren. Der Checker zielt darauf, allgemeine Fehler im Umgang mit Bibliotheksfunktionen zu finden, so wie strlcpy(3) oder sscanf(3), ohne irrtümliche Positivmeldungen auszugeben. Die Anwendungen auf den Source- und Ports-Tree zeigte über hundert echte Bugs auf, die wo immer möglich ausgebessert und den ursprünglichen Autoren zurückgeschickt wurden.

  • Privilege separation wurde für den syslogd(8) daemon implementiert, wodurch er widerstandsfähiger gegen zukünftige Fehler wurde. Der Kindprozeß, der am Netzwerk lauscht, läuft nun als normaler User und chroot'et sich selbst, während der Elternprozeß den Status des Kinds verfolgt und an seiner Stelle privilegierte Operationen ausführt.

  • Viele unsichere String-Funktionen wurden aus dem Kernel und aus Userland-Programmen entfernt. Dieses Audit ist eines der umfassendsten, das OpenBSD je unternommen hat, mit tausenden Vorkommnissen von strcpy(3), strcat(3), sprintf(3) und vsprintf(3), die durch sicherere, beschränkte Alternativen ersetzt wurden wie strlcpy(3), strlcat(3), snprintf(3), vsnprintf(3) und asprintf(3).

  • Viele Verbesserungen und behobene Fehler im ProPolice stack protector. Mehrere andere Fehler bei der Codegenerierung für RISC Architekturen behoben.

  • ProPolice stack protection wurde nun auch im Kernel aktiviert.

  • Privilege separation wurde in den X Server implementiert. Der privilegierte Kindprozeß ist für die Operationen verantwortlich, die nicht mehr ausgeführt werden können, nachdem der Hauptprozeß zu einem nichtprivilegierten User umgeschaltet wurde. Dies reduziert den möglichen Schaden durch böswillige X Clients, im Falle eines Bugs im X Server, stark.

  • Unterstützung für die Binärkompatibilität wird nun über sysctl(8) kontrolliert. Die Emulation ist nun per default ausgeschaltet, um die Angriffsfläche für böswillige Binaries zu begrenzen und kann in sysctl.conf(5) eingeschaltet werden.

  • Die Manual Pages wurden stark aufgeräumt und verbessert.

  • Der Ports-Tree unterstützt nun das Bauen von Programmen unter systrace(1), und verringert damit die Wahrscheinlichkeit von Anwendungen, die das System zur Zeit der Kompilation mittels trojanisierter Konfigurationsskripte o. ä. beschädigen.

  • Symbol caching in ld.so(1) verringert die Startdauer großer Anwendungen.

  • Mehr Lizenzausbesserungen, unter anderem die Entfernung der Werbeklausel in großen Teilen des Source-Trees.

  • Ersetzung der GNU Kommandos diff(1), diff3(1), grep(1), egrep(1), fgrep(1), zgrep(1), zegrep(1), zfgrep(1), gzip(1), zcat(1), gunzip(1), gzcat(1), zcmp(1), zmore(1), zdiff(1), zforce(1), gzexe(1) und znew(1) durch Gegenstücke unter BSD-Lizenz.

  • Unterstützung für read-only NTFS Dateisysteme hinzugefügt.

  • Verbesserungen bei der Zuverlässigkeit überlagerter Dateisysteme, was NULLFS wieder produktionsfähig gemacht hat.

  • Import des growfs(8) Tools, das die Vergrößerung existierender Dateisysteme erlaubt.

  • Verbesserungen bei der Linux Emulation erlauben es nun, noch mehr Anwendungen laufen zu lassen.

  • Bemerkenswerte Verbesserungen in der pthreads(3) Bibliothek.

  • Ersetzung vieler statischer Verwendungen von fd_set durch poll(2) oder dynamische Allokation.

  • ANSI-fikation und striktere Prototypen für große Teile des Source-Trees.

  • Alte KerberosIV Unterstützung wurde entfernt, und die verbleibende KerberosV Codebasis wurde für leichtere Handhabung umstrukturiert.

  • Über 2400 Ports, 2200 bereits gebaute Pakete.

  • Große Zahl an Fehlerkorrekturen, Änderungen und Optimierungen bei unserem Paketfilter pf(4), unter anderem:
    • packet tagging (z. B. Filter auf Tags, die basierend auf der MAC Adresse durch die Bridge hinzugefügt wurden)
    • stateful TCP normalization (verhindert Berechnung der Uptime und Erkennung von NAT)
    • passive OS detection (Basierend auf dem OS können Verbindungen gefiltert oder umgeleitet werden)
    • SYN proxy (schützt Server gegen SYN flood Attacken)
    • adaptive state timeouts (verhindert Überläufe in der Zustandstabelle bei Angriffen)

  • Verbesserte Hardwareunterstützung, unter anderem:
    • Kauai ATA Controller (Apple ATA100 wdc) kauaiata(4) unterstützt Powerbook 12" und 17" Modelle.
    • Unterstützung für die Kontrolle der LongRun Register der Transmeta CPUs.
    • Viele Fehler behoben bei den aac(4), ahc(4), osiop(4) und siop(4) SCSI Treibern.
    • Neue it(4), lm(4) und viaenv(4) Hardwaremonitor Treiber
    • Neuer safe(4) Treiber fü SafeNet Kryptobeschleuniger.
    • Neuer mtd(4) Treiber für Myson Technologies Netzwerkkarten.
    • Mehr unterstützte Ethernet Karten durch sk(4), wi(4), fxp(4) und dc(4).
    • Gründliche Überholung und Synchronisation des kompletten usb(4) Subsystems mit NetBSD.
    • Neue und bessere Unterstützung für verschiedene Controller in pciide(4), unter anderem experimentelle Unterstützung für Serial ATA.
    • Neue Treiber, um die mgx(4) and pninek(4) SPARC Framebuffer zu unterstützen. Der vigra(4) Treiber unterstützt ebenfalls mehrere Modelle.
    • pcmcia(4) Unterstützung für Tadpole SPARCBooks und SPARCs mit PCMCIA-SBus Bridges.
    • Watchdog Unterstützung für elansc(4) und geodesc(4), wie sie auf Soekris Boards verwendet werden.

  • Das System schließt folgende große Komponenten von Dritten mit ein:
    • XFree86 4.3.0 (+ patches, und i386 enthält auch 3.3.X Server, deshalb Unterstützung aller Chipsätze)
    • Gcc 2.95.3 (+ patches)
    • Perl 5.8.0 (+ patches)
    • Apache 1.3.28, mod_ssl 2.8.15, DSO support (+ patches)
    • OpenSSL 0.9.7b (+ patches)
    • Groff 1.15
    • Sendmail 8.12.9 (+ parse8.359.2.8 security patch)
    • Bind 9.2.2 (+ patches)
    • Lynx 2.8.4rel.1 mit HTTPS und IPv6 support (+ patches)
    • Sudo 1.6.7p5
    • Ncurses 5.2
    • Latest KAME IPv6
    • Heimdal 0.6rc1 (+ patches)
    • Arla-current
    • OpenSSH 3.7.1 (jetzt mit GSSAPI Unterstützung)

  • Viele Verbesserungen in Sicherheit und Zuverlässigkeit (beachte die rotgedruckten Stellen im kompletten Changelog).

  • und viel mehr.

Wie man es installiert

Es folgen Anweisungen, die Du auf gedrucktem Papier hättest, wenn Du ein CDROM Set gekauft hättest, anstatt eine Alternativform der Installation zu benutzen. Die Anweisungen für eine Installation per FTP (oder eine andere Form) sind sehr ähnlich; die Anweisungen für CDROM wurden belassen, damit Du sehen kannst, wieviel einfacher es gewesen wäre, stattdessen eine CD zu kaufen.

Bitte beachte auch die folgenden Dateien auf den drei CDROMs oder den FTP-Servern, die noch weitere Details enthalten, um OpenBSD 3.4 auf deiner Maschine zu installieren:

  • CD1:3.4/i386/INSTALL.i386

  • CD2:3.4/macppc/INSTALL.macppc
  • CD2:3.4/vax/INSTALL.vax

  • CD3:3.4/sparc/INSTALL.sparc
  • CD3:3.4/sparc64/INSTALL.sparc64

  • FTP:.../OpenBSD/3.4/alpha/INSTALL.alpha
  • FTP:.../OpenBSD/3.4/hp300/INSTALL.hp300
  • FTP:.../OpenBSD/3.4/hppa/INSTALL.hppa
  • FTP:.../OpenBSD/3.4/mac68k/INSTALL.mac68k
  • FTP:.../OpenBSD/3.4/mvme68k/INSTALL.mvme68k

Hier gibt es kurze Installationsinformationen für Leute, die mit OpenBSD vertraut sind und den Befehl "disklabel -E" kennen. Wenn du irgendwie unsicher in Bezug auf die Installation bist, lies die entsprechenden INSTALL.* Dateien, die oben angegeben sind!

OpenBSD/i386:

    Spiele mit den BIOS-Optionen herum und versuche, von CD zu booten. Die i386 Version von OpenBSD ist auf der CD1. Wenn dein BIOS kein Booten von CD unterstützt, schreibe CD1:3.4/i386/floppy34.fs auf eine Floppy, und nimm diese als Boot-Medium.

    Benutze CD1:3.4/i386/floppyB34.fs für eine bessere Unterstützung von SCSI-Controllern oder CD1:3.4/i386/floppyC34.fs für bessere Laptop-Unterstützung.

    Wenn du OpenBSD neben anderen Betriebssystemen betreiben willst, solltest du unbedingt das Dokument INSTALL.i386 lesen.

    Um eine Bootdiskette unter MS-DOS zu erstellen, benutze das "rawrite" Programm, das sich unter CD:/3.4/tools/rawrite.exe befindet. Unter einem Unix-artigen Betriebssystem wird dd(1) verwendet. Hier ist ein Beispiel für dd(1) , wobei das Gerät device durch "floppy", "rfd0c" oder "rfd0a" ersetzt wird. 

      # dd if=<file> of=/dev/<device> bs=32k

    Benutze ausschließlich perfekte und sauber formatierte Floppies OHNE DEFEKTE BLÖCKE oder deine Installation wird wahrscheinlich nicht klappen. Mehr Informationen über die Installation von OpenBSD/i386 gibt es in der FAQ4.1.

OpenBSD/macppc:

    Lege die CD2 in dein CDROM Laufwerk und drücke die C Taste während des Einschaltens, bis schließlich die Anzeige OpenBSD/macppc boot erscheint.

    Alternativ kannst du am Open Firmware prompt boot cd:,ofwboot /3.3/macppc/bsd.rd eingeben.

OpenBSD/vax:

    Boote über das Netzwerk via mopbooting, so wie es in INSTALL.vax beschrieben ist.

OpenBSD/sparc:

    Version 3.4 von OpenBSD/sparc befindet sich auf der CD3. Um von dieser CD zu booten, kannst du eines von den zwei Kommandos benutzen, die weiter unten aufgeführt sind, je nachdem welche ROM Version du hast. 
      > boot cdrom 3.4/sparc/bsd.rd
or
> boot sd(0,6,0)3.4/sparc/bsd.rd

    Wenn deine Sparc kein CD Laufwerk hat, kannst du alternativ dazu auch von einer Floppy booten. Dazu musst du die Datei "CD3:3.4/sparc/floppy34.fs" auf eine Floppy schreiben. Mehr Informationen gibt es in der FAQ4.1. Um von dieser Floppy zu booten, kannst du eines von den zwei Kommandos benutzen, die weiter unten aufgeführt sind, je nachdem welche ROM Version du hast. 

      > boot floppy
oder
> boot fd()

    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

    Wenn deine Sparc weder Floppy noch CD-Laufwerk hat, kannst du entweder ein bootbares Band erzeugen, oder via Netzwerk installieren, indem du den Anweisungen in der Datei INSTALL.sparc folgst.

OpenBSD/sparc64:

    Lege die CD3 in dein CDROM Laufwerk und gib boot cdrom ein.

    Wenn das nicht klappt oder du kein CDROM-Laufwerk hast, kannst du die Datei CD3:343/sparc64/floppy34.fs auf eine Floppy schreiben und sie mit boot floppy booten.
    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

    Du kannst auch CD3:3.4/sparc64/miniroot34.fs auf die Swap-Partition auf der Platte schreiben und mit boot disk:b booten.

    Wenn nichts davon klappt, kannst du auch vom Netz booten, wie es in. INSTALL.sparc64 beschrieben ist.

OpenBSD/alpha:

    Schreibe FTP:3.4/alpha/floppy34.fs oder FTP:3.4/alpha/floppyB34.fs (abhängig von deiner Maschine) auf eine Diskette und gib boot dva0 ein. INSTALL.alpha enthält dazu weitere Details.

    Stelle sicher, dass du eine sauber formatierte Floppy OHNE DEFEKTE BLÖCKE benutzt, ansonsten wird deine Installation mit hoher Wahrscheinlichkeit nicht klappen.

OpenBSD/hp300:

    Boote mittels der Anweisungen in INSTALL.hp300 über dein Netzwerk.

OpenBSD/hppa:

    Boote vom Netzwerk, indem du die Anweisungen in INSTALL.hppa oder von der hppa Plattform Seite befolgst.

OpenBSD/mac68k:

    Boote MacOS ganz normal und partitioniere deine Festplatte mit den normalen A/UX Konfigurationen. Dann entpacke die Macside Utilities aus FTP:3.4/mac68k/utils auf deine Festplatte. Benutze Mkfs, um deine Dateisysteme auf den A/UX Partitionen, die du gerade erzeugt hast, zu erstellen. Dann benutze den "BSD/Mac68k Installer", um alle Sets in FTP:3.4/mac68k/ auf deine Partitionen zu kopieren. Letzlich bist du nun soweit, dass du den "BSD/Mac68k Booter" mit dem Ort, an dem dein Kernel liegt, konfigurieren kannst, und dann das System booten kannst.

OpenBSD/mvme68k:

    Du kannst ein bootbares Installationsband erzeugen, oder über Netzwerk booten.
    Der Netzwerk Boot benötigt eine MVME68K BUG Version dass die NIOT und NBO Debugger Kommandos unterstützt. Die Anweisungen in INSTALL.mvme68k beinhalten weitere Details.

Hinweise zum Quellcode

    src.tar.gz enthält ein Source-Code-Archiv, das in /usr/src beginnt. Diese Datei enthält bis auf die Kernel-Sourcen, die sich in einem anderen Archiv befinden, alles, was du brauchst. Um es zu entpacken: 

      # mkdir -p /usr/src
# cd /usr/src
# tar xvfz /tmp/src.tar.gz

    sys.tar.gz enthält ein Quellcode-Archiv, das in /usr/src/sys beginnt. Diese Datei enthält alle Kernel-Quellen, die man benötigt, um einen neuen Kernel zu erzeugen. Um sie zu entpacken: 

      # mkdir -p /usr/src/sys
# cd /usr/src
# tar xvfz /tmp/sys.tar.gz

    Beide Archive enthalten einen regulären CVS checkout. Wenn man diese Dateibäume benutzt, ist es möglich, die anoncvs-Server wie hier beschrieben zu benutzen. Wenn man diese Dateien benutzt, hat man ein viel schnelleres erstes Update, als wenn man einen frischen, eigenen OpenBSD Quellbaum (source tree) mit Hilfe von CVS holt.

Ports Tree

Mitgeliefert wird auch ein ports tree Archiv. Um es zu entpacken: 

    # cd /usr
# tar xvfz /tmp/ports.tar.gz
# cd ports

Das ports/ Unterverzeichnis ist ein checkout für den OpenBSD ports tree. Lies am besten die Ports Seite, wenn du noch nichts über Ports weisst. Dieser Text ist keine Anleitung, wie man die Ports benutzt, sondern nur dazu da, um den Leuten den Einstieg in das OpenBSD ports System zu ermöglichen.

Das ports/ Verzeichnis stellt ein CVS (siehe in der manpage für cvs(1) nach, wenn du mit CVS nicht vertraut bist) checkout unserer ports dar. Wie unser kompletter source tree ist auch auch unser ports tree via anoncvs verfügbar. Um ihn also auf dem Laufenden zu halten, musst du den ports/ tree auf einem schreibbaren Medium haben und den tree mit diesem Kommando updaten: 

    # cd [portsdir]/; cvs -d anoncvsserver.openbsd.org:/cvs update -Pd -rOPENBSD_3_4

[Natürlich musst du das lokale Verzeichnis und den Servernamen durch den Ort deiner Ports-Collection und des verwendeten AnonCVS-Servers ersetzen.]

Denk daran, dass die meisten ports via ftp als fertige packages erhältlich sind. Upgedatete packages für 3.4 werden verfügbar gemacht, sobald Probleme bekannt sind.

Falls du gerne einen weiteren port aufgenommen sehen würdest, mithelfen willst oder einfach mehr erfahren willst, solltest du die Mailingliste ports@openbsd.org kennenlernen.
